# Consulting Toolkit (컨설팅툴킷)

컨설팅 5계열 25기술(MECE=미씨, 피라미드원칙, 소크라테스질문, 5Whys 등) 자동 라우팅. 형은 자연어만 쓰고, Claude가 적합한 컨설팅기술을 꺼내 적용한다.

---

## 절대 규칙

| # | 규칙 | 이유 |
|---|------|------|
| 1 | **의도 파싱 후 기술 선언 필수** — 출력 시작 시 "적용 기술: {계열}.{기술명}" 1줄 명시 | 형이 어떤 기술이 쓰였는지 알아야 학습·재사용 가능 |
| 2 | **2기술 이상 조합 시 순서 고정** — 구조화 → 문제해결 → 질문 → 설득 → 실행 순 (역류 금지) | 맥킨지 표준 흐름. 역류하면 논리 꼬임 |
| 3 | **24개 사고도구(홈즈·오컴 등)와 겹치면 trigger-dictionary로 라우팅** | 영역 중복 방지. 컨설팅 테크닉 ≠ 사고도구 |
| 4 | **확신도 병기** — 각 기술 적용 결과에 높음90·보통70·낮음50 + 근거 1줄 (UP v37.1 §1 준수) | 가설을 사실처럼 말하지 않기 |
| 5 | **근거 없으면 "모름" 선언** — 기술이 요구하는 데이터 부재 시 해당 단계 스킵 + 명시 | 패턴완성·환각 차단 |

## 실행 흐름

```
① 의도 파싱 → ② 계열 라우팅 → ③ 기술 적용 → ④ 결과 통합
```

### ① 의도 파싱 (30초)

형 입력에서 **동사/명사/상황** 3요소 추출:

| 추출 신호 | 의미 | 라우팅 후보 |
|---|---|---|
| "쪼개/분해/정리/구조화" | 생각 정리 | §A. 구조화 계열 |
| "왜/근본/원인/진짜 이유" | 답 찾기 | §B. 문제해결 계열 |
| "물어봐/인터뷰/캐물어/다시 정의" | 정보 발굴 | §C. 질문 계열 |
| "설득/보고/전달/한 장으로" | 설득 | §D. 커뮤니케이션 계열 |
| "합의/실행/워크샵/변화" | 움직임 유도 | §E. 개입 계열 |

**1개 계열만 매칭**: 해당 계열 기본 기술 1~2개 적용
**2개+ 계열 매칭**: 흐름 순서대로 순차 적용 (규칙 #2)

### ② 계열 라우팅 (매칭 테이블)

**상세:** → references/technique-catalog.md 참조

간단 매칭:

| 계열 | 핵심 기술 | 언제 |
|------|----------|------|
| A. 구조화 | MECE, 이슈트리, 피라미드, SCQA, 로직트리 | 문제가 모호할 때 |
| B. 문제해결 | 가설검증, RCA, 5Whys, 파레토, 제1원리 | 원인·해법 찾을 때 |
| C. 질문 | 소크라테스, Funneling, Reframing, 액티브리스닝, Probing | 정보 부족할 때 |
| D. 커뮤니케이션 | Storylining, Data Storytelling, Exec Summary, Visualization, Stakeholder Mapping | 설득·보고할 때 |
| E. 개입 | Workshop Facilitation, Alignment, Decision Framework, Change Mgmt, Nudging | 실행·합의 필요할 때 |

### ③ 기술 적용

**상세 프로토콜:** → references/technique-catalog.md 참조

각 기술은 **입력→처리→출력** 3단으로 정형화. 예시:

- **MECE**: 입력=대상, 처리=중복·누락 점검, 출력=2~4개 카테고리
- **5Whys**: 입력=현상, 처리=왜×5, 출력=근본원인 1개
- **SCQA**: 입력=보고주제, 처리=Situation→Complication→Question→Answer, 출력=도입부
- **Pyramid**: 입력=결론, 처리=MECE 근거 3개, 출력=피라미드 구조

### ④ 결과 통합

출력 상단에 **반드시** 포함:

```
적용 기술: {계열}.{기술1} → {계열}.{기술2}
확신도: {높음90/보통70/낮음50} 근거: {1줄}
```

## 3가지 사용 모드

| 모드 | 트리거 | 출력 |
|------|--------|------|
| 진단 | "진단해줘", "어떤 기술 써야 돼?" | 적합 기술 3개 추천 + 이유 |
| 적용 | 자연어 의도 (기본) | 기술 자동 선택 + 결과 |
| 설계 | "워크플로우 짜줘", "{프로젝트} 어떻게 접근해?" | 다단계 기술 체인 설계 |

## 예시

**입력:** "이 사업이 왜 매출이 안 오르는지 분석해줘"

**파싱:** "왜 안 오르는지" = 문제해결 계열 + "분석" = 구조화 보조

**적용 기술:** A.이슈트리 → B.5Whys → B.파레토

**출력:**
```
적용 기술: 구조화.이슈트리 → 문제해결.5Whys → 문제해결.파레토
확신도: 보통70 근거: 데이터 부재, 가설 기반

1. 이슈트리로 매출 감소 요인 3갈래 분해
2. 각 갈래에 5Whys로 근본원인 추적
3. 파레토로 80/20 우선순위
```

## 기존 스킬과 경계

| 스킬 | 역할 | consulting-toolkit과 차이 |
|------|------|--------------------------|
| trigger-dictionary | 24개 사고도구(홈즈·오컴·프리모르템) | 철학/프레임 기반 vs 본 스킬은 실무 방법론 |
| biz-skill | 18축 사업전략 패턴 매칭 | 사업 도메인 특화 vs 본 스킬은 도메인 무관 |
| meeting-engine | 회의 설계·진행·기록 | 회의 전주기 vs 본 스킬은 퍼실리테이션 기술 1개만 차용 |
| negotiation-skill | 5-Layer 협상 | 협상 전문 vs 본 스킬은 Stakeholder Mapping 수준 |
| paper-engine | 산출물 품질 관리 | 문서 구조 vs 본 스킬은 설득 기술 |

**충돌 시 우선순위:** 본 스킬의 NOT 라우팅이 작동. 형이 "협상해줘"라고 하면 negotiation-skill로 넘김.

## Gotchas

| 함정 | 대응 |
|------|------|
| 5계열 모두 적용하려 함 | 1~3개만 선택. "컨설팅 전체 쇼" 금지 |
| 기술 이름만 나열 후 내용 없음 | 각 기술의 입력·처리·출력을 **실제로** 수행 |
| 형의 데이터가 없는데 기술 적용 | 규칙 #5 — "모름" 선언하고 데이터 요청 |
| 트리거 단어 외우기 강요 | 절대 금지. 자연어만으로 작동해야 함 (본 스킬의 존재 이유) |
| trigger-dictionary 24개 사고도구와 중복 호출 | P1 NOT 라우팅으로 차단. 사고도구는 철학·프레임 / 본 스킬은 실무 방법론 |
| **중복 4쌍 경계** (제1원리·넛지·백본·스켈레톤) | 트리거어 **단어 명시** 호출("제1원리 해줘"·"넛지 설계") → trigger-dictionary 우선. **자연어 의도** ("기존 방식 안 돼"·"행동 유도") → consulting-toolkit 발동. 같은 도구여도 진입점 다름 |
| 기술 적용 선언 빠뜨림 | 규칙 #1 위반. 출력 상단에 "적용 기술:" 1줄 필수 |
