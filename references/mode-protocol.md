# Mode Protocol

3가지 사용 모드의 입·출력 정형. 모드 자동 판별 + 명시 선언 포함.

---

## 모드 자동 판별

| 입력 신호 | 판별 | 모드 |
|-----------|------|------|
| "어떤 기술?", "뭘 써야 돼?", "추천", "접근법" | 진단 요청 | 진단 |
| 자연어 의도 (기본값) | 암시적 적용 | 적용 |
| "워크플로우", "다단계", "프로젝트 접근", "전체 설계" | 체인 요청 | 설계 |
| 명시적 "{모드명} 모드로" | 강제 지정 | 해당 모드 |

---

## 진단 모드

**목적:** 적합한 기술 추천 (아직 실행 X)

**입·출력 정형:**
```
[입력] 형의 상황·목적·제약

[출력]
## 진단 결과

상황 요약: {1줄}
핵심 제약: {시간/데이터/조직/기술} 중 걸리는 것

추천 기술 3선 (우선순위):

1순위: {계열}.{기술} + {보조기술}
  이유: {왜 이 기술이 1순위인지 2줄}

2순위: {기술}
  이유: {2줄}

3순위: {기술}
  이유: {2줄}

선택해서 "N번으로 가자" 하면 적용 모드 진입.
```

**체크:**
- [ ] 추천 3개 고정 (2개·4개 X)
- [ ] 각 추천에 "왜" 포함
- [ ] 형의 선택 요청으로 마무리

---

## 적용 모드 (기본)

**목적:** 기술 자동 선택 + 실행

**입·출력 정형:**
```
[입력] 자연어 의도 (예: "왜 매출 떨어지는지 분석해")

[출력]
적용 기술: {계열}.{기술1} → {계열}.{기술2}
확신도: {높음90/보통70/낮음50} 근거: {1줄}

{기술1 결과}
{기술2 결과}

{다음 단계 제안 1줄}
```

**체크:**
- [ ] 출력 상단 "적용 기술:" 1줄 (규칙 #1)
- [ ] 확신도 + 근거 (UP v37.1 §1)
- [ ] 데이터 부재 시 "모름" 선언 (규칙 #5)

---

## 설계 모드

**목적:** 프로젝트 전체를 다단계 기술 체인으로 설계

**입·출력 정형:**
```
[입력] 프로젝트·큰 목표 (예: "신사업 기획 어떻게 접근?")

[출력]
## 워크플로우 설계

프로젝트: {1줄 요약}
기간·단계: {예: 4주 / 4 Phase}

Phase 1: {이름}
  기술: {계열}.{기술}
  산출물: {무엇}

Phase 2: {이름}
  기술: ...
  산출물: ...

Phase 3: ...
Phase 4: ...

## 주의사항
- {각 Phase 전환 시 함정}
- {자원 고려사항}
```

**체크:**
- [ ] Phase 3~5개 (과도하면 실행 불가)
- [ ] 각 Phase에 기술·산출물 명시
- [ ] 규칙 #2 순서 준수: 구조화→문제해결→질문→설득→실행

---

## 모드 전환 규칙

| 전환 | 트리거 | 동작 |
|------|--------|------|
| 진단 → 적용 | 형이 "N번으로 가자" | 선택한 기술 적용 |
| 적용 → 진단 | 중간에 "다른 방법은?" | 현재 접근 일시정지 + 재진단 |
| 적용 → 설계 | 결과가 프로젝트급이라 판단 | "이건 여러 단계 필요합니다. 설계 모드로 전환할까요?" 확인 |
| 설계 → 적용 | 형이 "Phase N 먼저 해줘" | 해당 Phase의 기술만 실행 |

---

## Gotchas

| 함정 | 대응 |
|------|------|
| 진단 모드에서 바로 적용까지 | 금지. 진단은 추천만. 형의 선택 기다림 |
| 적용 모드에서 5개+ 기술 체인 | 과잉. 1~3개로 수렴 |
| 설계 모드에서 Phase 7+ | 실행 불가. 3~5개로 압축 |
| 모드 자동 판별 실패 시 | 형에게 1줄로 확인: "진단 / 적용 / 설계 중 어느 쪽?" |
