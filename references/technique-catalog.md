# Consulting Technique Catalog

5계열 × 5기술 = 25기술. 각 기술은 **입력→처리→출력** 3단 정형.

---

## A. 구조화 계열 (Thinking Frame)

### A1. MECE (Mutually Exclusive, Collectively Exhaustive)
- **입력:** 분해 대상 (문제·시장·고객 등)
- **처리:** 중복(ME)·누락(CE) 없이 2~4개 카테고리로 분해. 각 카테고리 교집합=0, 합집합=전체
- **출력:** 카테고리 리스트 + 각 카테고리 정의 1줄
- **언제:** 이슈 분해의 첫 단계. 다른 구조화 기술의 기반
- **함정:** 7개+ 분해 시 MECE 붕괴. 2~4개가 최적

### A2. Issue Tree (이슈트리)
- **입력:** 큰 질문 (예: "왜 매출이 떨어졌나?")
- **처리:** MECE로 2~3 depth 분해. 루트→중간이슈→세부이슈
- **출력:** 트리 다이어그램 또는 들여쓰기 리스트
- **언제:** 문제가 복잡하고 여러 원인 가능성이 있을 때
- **함정:** depth 4+ 가면 분석 마비. 3 depth 이내 유지

### A3. Pyramid Principle (피라미드 원칙, Barbara Minto) ⚠️ trigger-dictionary '백본'과 인접
- **입력:** 결론 1문장
- **처리:** 결론 아래 MECE 근거 3개, 각 근거 아래 팩트 3개
- **출력:** 3층 피라미드. Top-down 서술
- **언제:** 임원 보고, 제안서 서두
- **함정:** Bottom-up(데이터부터)로 시작하면 결론 산만

### A4. SCQA (Situation-Complication-Question-Answer)
- **입력:** 보고·제안 주제
- **처리:** S(배경)→C(변화·문제)→Q(그래서 뭐?)→A(결론) 4단 구성
- **출력:** 도입부 4문단 또는 슬라이드 1장
- **언제:** 문서·프레젠테이션 오프닝
- **함정:** C(Complication) 약하면 독자 몰입 실패. "그래서 뭐가 문제냐"가 명확해야

### A5. Logic Tree (로직트리)
- **입력:** 해결해야 할 질문
- **처리:** Why-tree(원인)/How-tree(해결) 선택 후 MECE 분해
- **출력:** Why 또는 How 트리
- **언제:** 원인 분석(Why) 또는 해결책 탐색(How)
- **함정:** Why와 How를 섞으면 트리가 꼬임. 하나만 선택

## B. 문제해결 계열 (Problem Solving)

### B1. Hypothesis-Driven (가설 기반)
- **입력:** 해결할 문제
- **처리:** 가설 3개 설정 → 각 가설 검증 방법 설계 → 빠른 검증 → 지지/기각 판정
- **출력:** 가설 테이블 (가설·검증법·결과·판정)
- **언제:** 시간 부족 + 모든 가능성 탐색 불가능할 때
- **함정:** 가설을 사실처럼 다루면 확증편향. 반증 데이터 먼저 찾기

### B2. Root Cause Analysis (RCA)
- **입력:** 현상 (증상)
- **처리:** 증상→직접원인→중간원인→근본원인 단계적 추적. 증거 필수
- **출력:** 근본원인 1~2개 + 증거 체인
- **언제:** 반복 발생 문제. 표면 대응으로 안 풀릴 때
- **함정:** "사람 탓"으로 귀결되면 대개 실패. 시스템·프로세스 원인 탐색

### B3. 5 Whys
- **입력:** 현상 1개
- **처리:** "왜?"를 5회 반복. 각 답이 다음 질문의 대상
- **출력:** Why 체인 5단 + 근본원인
- **언제:** RCA의 간이판. 5분 안에 원인 추적
- **함정:** 3회 안에 "사람 문제"로 가면 편향. 구조·프로세스로 파고

### B4. Pareto (80/20)
- **입력:** 다수 요인 (10개+)
- **처리:** 임팩트 기준 정렬 → 상위 20%가 결과 80% 차지하는지 확인
- **출력:** 우선순위 리스트 (상위 20% = 핵심)
- **언제:** 리소스 제한 + 어디에 집중할지 결정
- **함정:** 데이터 없이 감으로 80/20 주장 = 오류. 측정 먼저

### B5. First Principles (제1원리) ⚠️ "제1원리" 단어명시=trigger-dictionary 우선
- **입력:** 당연시되는 전제
- **처리:** 전제 해체 → "진짜 물리·수학·근본 법칙만 남기기" → 재구성
- **출력:** 기존 통념과 다른 재구성된 해법
- **언제:** 혁신·돌파 필요. 업계 통념이 한계일 때
- **함정:** 시간·비용 크게 듦. 일상 의사결정에는 과잉

## C. 질문·인터뷰 계열 (Discovery)

### C1. Socratic Questioning (소크라테스 질문)
- **입력:** 주장·전제
- **처리:** "왜 그렇게 생각하세요?" "반대 증거는?" "전제가 바뀌면?" 계층 질문
- **출력:** 숨은 전제·논리 구멍 발견
- **언제:** 상대의 주장을 검증·해체할 때
- **함정:** 공격적으로 느껴질 수 있음. 톤 관리 필수

### C2. Funneling (깔때기)
- **입력:** 인터뷰 목적
- **처리:** 넓은 개방형 질문 → 좁은 구체형 질문 순
- **출력:** 인터뷰 스크립트 (Top→Bottom)
- **언제:** 고객 인터뷰, 전문가 인터뷰
- **함정:** 처음부터 구체질문 = 방어적 답변 유발

### C3. Reframing (재정의)
- **입력:** 기존 문제 정의
- **처리:** "이 문제를 다르게 보면?" 시점·주어·범위 변경
- **출력:** 재정의된 문제 진술
- **언제:** 답이 안 나올 때 = 문제가 틀렸을 때
- **함정:** 너무 자주 재정의하면 논의가 겉돔. 1~2회만

### C4. Active Listening (액티브 리스닝)
- **입력:** 상대 발언
- **처리:** 요약 반영 ("즉 ~라는 말씀") + 감정 반영 ("답답하신 거군요")
- **출력:** 상대가 "이해받았다" 느낌
- **언제:** 인터뷰·워크샵·갈등 상황
- **함정:** 요약만 하고 감정 놓치면 기계적. 둘 다 필요

### C5. Probing (파고들기)
- **입력:** 상대의 표면 답변
- **처리:** "예를 들면?" "구체적으로?" "그때 어땠어요?" 연쇄 심화
- **출력:** 표면 아래 숨은 정보
- **언제:** 상대가 추상적·방어적 답을 할 때
- **함정:** 취조처럼 느껴지면 역효과. 2~3회 이내

## D. 커뮤니케이션 계열 (Delivery)

### D1. Storylining (스토리라인)
- **입력:** 전달할 메시지 여러 개
- **처리:** Hero's Journey 또는 Problem-Solution-Benefit 구조로 배열
- **출력:** 흐름이 있는 슬라이드/문서 순서
- **언제:** 프레젠테이션, 제안서
- **함정:** 데이터 덤프 = 스토리 아님. 주인공(청중)의 변화가 있어야

### D2. Data Storytelling
- **입력:** 데이터 + 전달 목적
- **처리:** 핵심 인사이트 1개 선정 → 그것을 부각하는 차트 1~2개 → 내러티브 연결
- **출력:** "데이터 → 인사이트 → 시사점" 3단
- **언제:** 데이터 기반 의사결정 요청 시
- **함정:** 차트 5개+ 붙이면 독자 포기. 1 슬라이드 1 메시지

### D3. Executive Summary (임원 요약)
- **입력:** 긴 보고서
- **처리:** 결론·근거·리스크·액션 4요소를 1페이지로 압축 (Pyramid 원칙)
- **출력:** 1페이지 요약 (읽는 데 60초)
- **언제:** 임원·이사회 보고
- **함정:** 상세 덧붙이면 임원이 안 읽음. 냉정한 압축

### D4. Visualization (시각화)
- **입력:** 숫자·관계·프로세스
- **처리:** 유형별 최적 차트 선택 (비교=막대, 추세=선, 구성=파이/도넛, 분포=산점도)
- **출력:** 차트 1~2개
- **언제:** 텍스트로 전달 어려울 때
- **함정:** 3D 차트·치장=가독성 하락. 단순 2D 권장

### D5. Stakeholder Mapping (이해관계자 지도)
- **입력:** 프로젝트 관련 인물·조직
- **처리:** 2×2 매트릭스 (영향력↑↓ × 관심도↑↓) 배치 → 각 사분면 대응전략
- **출력:** 2×2 맵 + 사분면별 액션
- **언제:** 조직 변화·제안 통과 필요 시
- **함정:** "영향력 높은 사람"만 챙기고 실무자 무시 = 실행 단계에서 막힘

## E. 개입·퍼실리테이션 계열 (Execution)

### E1. Workshop Facilitation
- **입력:** 워크샵 목적·참석자
- **처리:** 아이스브레이킹→발산(브레인스토밍)→수렴(투표·우선순위)→액션 4단
- **출력:** 워크샵 결과물 + 액션리스트
- **언제:** 팀 합의·공동 설계 필요 시
- **함정:** 발산만 하고 수렴 안 하면 회의록만 남음. 수렴 필수

### E2. Alignment Building (합의 형성)
- **입력:** 의견 갈라지는 상황
- **처리:** 공통점 먼저 확인 → 차이점 명시 → 각자의 근거 수집 → 기준 합의 → 결정
- **출력:** 공통점·차이점·결정·근거 표
- **언제:** 팀/파트너 간 이견
- **함정:** 차이점부터 다루면 감정 고조. 공통점으로 시작

### E3. Decision Framework (의사결정 프레임)
- **입력:** 결정할 사안
- **처리:** 옵션 나열 → 평가 기준 설정 (3~5개) → 가중치 → 점수화 → 결정
- **출력:** 의사결정 매트릭스
- **언제:** 2개+ 옵션 중 선택 필요
- **함정:** 기준이 결과를 결정. 기준 설정에 시간 투자

### E4. Change Management (변화 관리, Kotter 8단계)
- **입력:** 변화 목표
- **처리:** 긴급감→연합→비전→공유→권한→단기성과→가속→정착 8단
- **출력:** 변화 로드맵
- **언제:** 조직 전환·신제도 도입
- **함정:** "긴급감" 없이 시작하면 동력 부족. 1단계가 가장 중요

### E5. Nudging (넛지, Thaler) ⚠️ "넛지" 단어명시=trigger-dictionary 우선
- **입력:** 유도할 행동
- **처리:** 기본값 설정·프레이밍·사회적 증거·즉시보상 중 1~2개 선택
- **출력:** 넛지 설계안
- **언제:** 강제 없이 행동 변화 유도
- **함정:** 조작으로 느껴지면 역효과. 투명성 유지

## Gotchas

| 함정 | 대응 |
|------|------|
| 25기술 다 쓰려고 함 | 1~3개만. 상황에 맞는 것만 |
| 기술 이름만 들먹이고 실제 적용 없음 | 입력·처리·출력 3단을 **실제로** 수행 |
| 데이터 없는데 B4.파레토 적용 | 규칙 #5 "모름" 선언 |
| C1.소크라테스로 공격적 질문 | 톤 관리. 검증이 목표지 논파가 아님 |
| 조합 체인 순서 거꾸로 | 규칙 #2 — 구조화→문제해결→질문→설득→실행 |
