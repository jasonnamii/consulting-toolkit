# Routing Examples

자연어 입력 → 기술 자동 선택 실제 예시 모음. 본 스킬의 핵심 가치(트리거 외우지 않기) 증명.

---

## 예시 1: 모호한 문제 분석

**입력:** "우리 앱 리텐션이 떨어지는데 뭐가 문제인지 모르겠어"

**파싱:**
- 신호: "떨어지는데" + "뭐가 문제" → 문제해결 + 구조화
- 데이터: 없음 (숫자·시점 미제공)

**적용 기술:** A2.이슈트리 → B3.5Whys

**출력:**
```
적용 기술: 구조화.이슈트리 → 문제해결.5Whys
확신도: 보통70 근거: 데이터 없음, 프레임만 제공

[이슈트리]
리텐션 하락
├── 획득 단계: 잘못된 유저가 들어옴
├── 활성화: 첫 경험 실패
└── 유지: 핵심가치 미전달

[5Whys: '첫 경험 실패' 가설]
1. 왜 첫 경험 실패? → 온보딩 이탈률 40%+
2. 왜 이탈? → 3단계 넘어가면 포기
3. 왜 3단계? → 가입 정보 과다
4. 왜 과다? → 마케팅 데이터 욕심
5. → 근본: '데이터 수집 vs UX'의 우선순위 미정의

[다음 액션]
실제 리텐션 데이터 제공해주면 파레토(B4) 붙여서 우선순위화
```

## 예시 2: 임원 보고 준비

**입력:** "다음주 이사회에 신사업 보고해야 되는데 한 장으로 정리하고 싶어"

**파싱:**
- 신호: "이사회" + "한 장" → 커뮤니케이션
- 추가: "보고" = 설득 구조 필요

**적용 기술:** A3.Pyramid → A4.SCQA → D3.Exec Summary

**출력:**
```
적용 기술: 구조화.Pyramid → 구조화.SCQA → 커뮤니케이션.Exec Summary
확신도: 높음90 근거: 정형 템플릿 적용

[SCQA 도입부]
S: {시장/내부 현재 상황 1줄}
C: {왜 지금 움직여야 하는지, 변화 요인}
Q: {그래서 우리는 뭘 해야 하나?}
A: {신사업 X를 Y 방식으로 런칭}

[Pyramid 본문]
결론: 신사업 X 런칭
├── 근거1 시장기회 (데이터 3개)
├── 근거2 내부역량 (데이터 3개)
└── 근거3 리스크 통제 (데이터 3개)

[Exec Summary 1페이지]
• 제안: {1줄}
• 근거: {3줄}
• 리소스: {1줄}
• 리스크·완화: {1줄}
• 의사결정 요청: {1줄}

→ 실제 숫자·내용 제공하면 채워서 완성
```

## 예시 3: 팀 갈등 해결

**입력:** "팀원들이 기술 스택 결정을 못 하고 싸우고 있어"

**파싱:**
- 신호: "못 하고" + "싸우고" → 개입·합의
- 추가: 옵션 비교 필요 → 의사결정 프레임

**적용 기술:** E2.Alignment → E3.Decision Framework

**출력:**
```
적용 기술: 개입.Alignment Building → 개입.Decision Framework
확신도: 높음85 근거: 정형 절차

[Alignment Step]
1. 공통점 확인: "모두 제품 성공을 원한다" (이견 없음)
2. 차이점 명시: A안(React) vs B안(Vue) - 각자 입장 1줄씩
3. 근거 수집: 각자 주장의 근거 3개씩 제시
4. 기준 합의: "우리는 뭘 기준으로 결정할까?" 먼저 합의

[Decision Framework Matrix]
기준 (가중치):
- 팀 숙련도 (30%)
- 생태계 크기 (25%)
- 성능 (20%)
- 유지보수성 (15%)
- 채용 용이성 (10%)

React vs Vue 각 기준 점수화(1~5) → 가중합 → 높은 쪽 선택

→ 팀에게 이 프레임을 공유하고 각자 점수 매겨오면 객관화됨
```

## 예시 4: 고객 인터뷰 설계

**입력:** "신제품 나오기 전에 잠재고객 인터뷰 어떻게 해야 돼?"

**파싱:**
- 신호: "인터뷰" + "어떻게" → 질문 계열

**적용 기술:** C2.Funneling → C4.Active Listening → C5.Probing

**출력:**
```
적용 기술: 질문.Funneling → 질문.Active Listening → 질문.Probing
확신도: 높음90 근거: 인터뷰 표준 프로토콜

[Funnel 구조 - 15~30분 인터뷰 기준]

Top (5분) 개방형
• "최근 {도메인}에서 가장 불편했던 경험은?"
• "하루를 어떻게 보내세요?"

Middle (10분) 반개방
• "그때 어떻게 해결하셨어요?"
• "다른 도구 써보신 적 있나요?"

Bottom (10~15분) 구체형
• "한 달에 이 문제로 몇 번 불편?"
• "얼마까지 내실 의향?"

[액티브 리스닝 체크]
- 요약 반영: "즉 ~이 가장 아쉬우셨다는 거죠?"
- 감정 반영: "많이 답답하셨겠어요"

[Probing 발동 조건]
상대가 "그냥" "별로" "대충" 대답 시 → "예를 들면?" "최근에 언제?"

→ 제품 도메인 알려주면 질문 리스트 구체화
```

## 예시 5: 혁신 아이디어 탐색

**입력:** "기존 방식으론 안 되겠어. 완전히 새로운 접근 찾고 싶어"

**파싱:**
- 신호: "기존 안 됨" + "새로운" → 혁신·돌파

**적용 기술:** B5.First Principles → A5.Logic Tree(How)

**출력:**
```
적용 기술: 문제해결.제1원리 → 구조화.로직트리(How)
확신도: 보통65 근거: 창의 기법, 결과 비결정적

[제1원리 분해]
1. 이 영역의 통념 3가지:
   - {통념1}
   - {통념2}
   - {통념3}

2. 각 통념 해체 — "이게 꼭 참인가? 물리·수학·근본 제약은 뭔가?"
3. 통념 제거 후 남는 근본 제약만 나열

[Logic Tree (How)]
새로운 접근 루트
├── 접근1: {근본제약 내에서 새로운 결합}
├── 접근2: {통념을 역전}
└── 접근3: {다른 도메인의 원리 이식}

→ 도메인·기존 방식을 구체적으로 알려주면 실제 분해 수행
```

## 예시 6: "어떤 기술 써야 돼?" (진단 모드)

**입력:** "연말까지 매출 2배 만들어야 되는데 어떤 기술로 접근할까?"

**모드:** 진단 (명시적 질문)

**출력:**
```
적용 기술 추천 3선:

1순위: B1.가설검증 + B4.파레토
  이유: 시간(연말) 제약 → 모든 옵션 탐색 불가. 가설 3개 설정하고
        파레토로 상위 20% 레버 찾기가 시간 대비 효율 최대

2순위: A2.이슈트리 + B3.5Whys
  이유: 매출 구성요소 분해(가격·수량·고객)로 어디가 병목인지 구조적 탐색

3순위: D5.Stakeholder Mapping + E4.Change Management
  이유: 매출 2배는 내부 변화 없이 불가. 조직 변화 동반 필요

선택 후 알려주시면 해당 기술로 실제 적용 들어감.
```

## Gotchas (예시 운영 시 함정)

| 함정 | 대응 |
|------|------|
| 예시가 입력과 100% 일치해야 한다고 생각 | 예시는 **패턴** 참고용. 실제 입력은 유연하게 매칭 |
| 매번 3기술 이상 체인 | 1기술로 끝나는 경우도 많음. 과잉 설계 금지 |
| 형의 데이터 없이 구체 숫자 생성 | 규칙 #5 "모름" 선언. 프레임만 제공 후 데이터 요청 |
